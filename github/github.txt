Git merge : bringing all chapters together. 

You and your friend writing a book.
you wrote 3 chapters
your friend wrote 2 chapters - at sametime. 
you both decided to combine for one final book.
--> If we use merge, it shows as you wrote 3 chapters, your frined wrote 2 chapters.

Git Rebase : Rewriting the story as one smooth timeline. 
You and your friend writing a book.
you wrote 3 chapters
your friend wrote 2 chapters - at sametime. 
you both decided to combine for one final book.
--> If we use rebase, it shows as you wrote 3 chapters first then your frined wrote 2 chapters for clean and lenear history.


--> use rebase when you want : "simplify the history", "The branch is only used by you" and To avoid unnessesary "merge commits" and to make it look like a straight story.

Golden Rule / Thumb Rule: Never rebase a branch that has been pushed to a shared/poublic repository. 

Why Rebasing Shared Branches Causes Chaos
--> It Changes Commit Hashes
--> It Destroys Teammates' Base History
--> It Multiplies Duplicate Commits
--> It Requires Dangerous Force Pushing

Summary Rule of Thumb:
Local-only branches: Rebase as much as you like to clean up your history before anyone else sees it.
Shared/Public branches: Always use git merge or git pull to integrate changes safely without breaking your team's workflow

git stash : 

Imagine you are working on some changes and suddenly your team member asked you to switch to another branch to fix a imp bug.

--> you can't commit yet, as your chnages are not ready, but you dont want to lose them.
so, stash your changes temp, then switch to other branch and fix it.. come back and unstash it.

git stash			--> stash the changes
git stash apply		--> Apply the stashed changes
git stash list		--> list the stashes
git stash show -p	--> show stash content
git stash drop		--> Drop a stash after apply
git stash push -m "msg"	--> create a stash with custom msg
git stash pop		-> apply and remove the stash
