S3 Storage Classes : 

--> How frequently you are accessing data.?
--> Do you need data without delay.? or okay to wait for sometime to get the data.?

S3 Standard : Frequently accessed data (more than once a month) with milliseconds access : 
--> We can access data without any delays. 
--> Data spans across mutliple AZs (>= 3 AZs)
--> 99.99% Availability.. 99.999999999% Durability.. 

Standard-IA : 
--> We can access data without any delays. 
--> Infrequently accessed data (once a month) with milliseconds access
--> Data spans across mutliple AZs (>= 3 AZs)
--> 99.99% Availability.. 99.999999999% Durability.. 

One Zone-IA : 
--> We can access data without any delays. 
--> As data stores only in 1 AZ, Recommended to store easily recreatable/reprodusable data. 
--> Infrequently accessed data (once a month) with milliseconds access
--> Data stores only in 1 AZ 
--> 99.95% Availability.. 99.999999999% Durability.. 

Glacier Instant Retrieval : 
--> Long-lived archive data accessed once a quarter with instant retrieval in milliseconds

Glacier Flexible Retrieval (formerly Glacier) : 
--> Long-lived archive data accessed once a year with retrieval of minutes to hours (2-5 hrs)
--> We cannot access data immediately. 

Glacier Deep Archive : 
--> Long-lived archive data accessed less than once a year with retrieval of hours. (12-24 hrs)
--> We cannot access data immediately. 

Intelligent-Tiering: 
--> Data with changing or unknown access patterns
--> Bit constier compare to other databases.

Glacier retrival options : 
Bulk retrieval: Typically within 5-12 hours.
Standard retrieval: Typically within 3-5 hours.
Expedited retrieval: Typically within 1-5 minutes when retrieving less than 250 MB.

By default, versioning will be in suspended state, existing objects wont have version id. 
After enableing versioning, All uplaods will get a version id. (we cannot manege the id)


--> We can get object to current state, if any wrong edits happens.
--> If any object deleted accidentally, we get a delete marker, delete the delete marker to get the object back.

If you enable "show version" and deleted an object, it deletes perminently.

Encryption : 

In-Transit encryption / At flight encryption : When the data is in travel state to s3 platform/webserver, Data will be encrypted. (webserver --> ssl/tls)

Client Side Encryption / CSE : Before uploading data to s3 platform, We can use our own encryption method and we can encrypt the data.. (Customer Responsibility)

Server Side Encryption / SSE : When data is in s3 platform, we can apply encry methods. We have multiple options.

Key Material (public key and private Key)

Server-side encryption with Amazon S3 managed keys (SSE-S3) : Default encryption method.. 
--> S3 genrates and manages the key material
--> Whoever has access to s3 platform, they can decrypt the data.. No addl permisisons required.
--> Suitable for public sharable data.. 
--> No direct access to key material to view/modify/delete/rotate.

---

SSE-KMS - DMK (Default Master Key)..
--> KMS Service generates and manages the key material
--> Whoever has access to s3 platform, they can decrypt the data.. No addl permisisons required.
--> No direct access to key material to view/modify/delete/rotate.
--> We cannot make object public.. 
--> THis key creates automatically, whenever we started using the s3 servcie. 

---

SSE-KMS CMK (Customer Managed Key)..
--> Customer has to Create the KMS Key and KMS Manages the key material.. We can use this key for multiple servcies.. 
--> Along with S3 service access, User/role should have kms key usage permissions to decrypt the data..
--> No direct access to key material to view/delete, Customer can rotate the key material. 
--> We cannot make object public.. 


---

SSE-KMS C (Customer Provided Key)..
--> Customer has to Create the KMS Key and Customer has to Manages the key material.. He need to upload the wrapped key to KMS servcie, so kms use this key to encrypt the data.. 
--> Along with S3 service access, User/role should have kms key usage permissions to decrypt the data..
--> Customer has access to key material to view/delete/rotate the key material. 
--> We cannot make object public.. 
--> https://www.youtube.com/watch?v=r17Z75TqjU0
