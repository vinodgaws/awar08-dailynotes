Without istio, 
Any microservice can talk to other microservice directly.
No visibility into who's talking to whom.
No control over encryption or traffic rules.

With Istio,
Every service gets a sidecar proxy (also called as envoy proxy), automatically injected.
All traffic flows through this proxy only.
We get security and traffic control without changing or improving our application code.

We dont use istio for basic or less microservices architectures, as this makes simple arch into complex.
Zero trust security : Every request will be validated and tested. 


1. Sidecar / emnvoy proxy : 
2. istiod / control plane : brain of istio..  (poilit (traffic rules), citadel (certificate) and galley (config validation))
3. mTLS / peerAuthentication : 
	3.1 STRICT: only mTLS is accepted, plain html will be rejected.
	3.2 Permissive : Both plain http and mtls are accepted.
4. Gateway : manages inbound traffix from outside the cluster. Similar to ingress controller, works with k8s gateway API.

--> Below command creates a container without sidecar, Am going to test the application access from a pod, which dont have sidecar/envoy proxy.. 

kubectl run curl-naked \
  -n default \
  --image=curlimages/curl:8.4.0 \
  --annotations='sidecar.istio.io/inject=false' \
  -- sleep 3600

kubectl exec -n default -it curl-naked -- \
  curl -v http://productpage:9080/productpage


---

If mTLS is STRICT → curl fails (503 or connection error)

If mTLS is PERMISSIVE → curl succeeds (200 OK) - Default mode.. From above pod, we are able to access the webpage, as mTLS defaulty setup as PERMISSIVE.

---
Now apply strict mTLS

apiVersion: security.istio.io/v1beta1
kind: PeerAuthentication
metadata:
  name: default
  namespace: default
spec:
  mtls:
    mode: STRICT
